# Decision Log

Key design and process decisions taken during the project, with the alternative considered, the
justification and the impact on requirements/tests. Decisions DEC-006 and DEC-007 result from the
change request (CR-01).

| ID | Decision | Alternative considered | Justification | Impact on requirements/tests |
|---|---|---|---|---|
| DEC-001 | Classify performance (REQ-007), security/authorization (REQ-008) and usability (REQ-009) as **non-functional** requirements. | Treat them as functional requirements. | They constrain the *quality* of existing behaviour (speed, who is allowed, ease of use), not new behaviour. NFRs must be measurable, so each got a concrete threshold/criterion. | Drove TC-007 (measure ≤3s) and TC-008 (role denial); shaped the AC to be testable. |
| DEC-002 | Split US-003 (Submit) into **US-003a (role authorization)** and **US-003b (completeness/validity)**. | Keep one large story. | Two independent concerns (who may submit vs whether data is valid). INVEST — independent, small, testable increments. | Enabled a dedicated role test (TC-008) and completeness/boundary tests (TC-004/006). |
| DEC-003 | Test the 90-day freshness rule with **boundary tests at 90 and 91 days** (TC-005, TC-006). | Test only a single "old" value. | Off-by-one errors (`>` vs `>=`) occur exactly at the limit; testing both sides catches them. | Concrete boundary values feed AT-003 and the seed data. |
| DEC-004 | In the **baseline**, stale critical evidence (>90 days) **blocks** submission. | Only flag stale evidence without blocking. | Q-007 (elicitation) left block-vs-flag open; blocking is the safer default for a readiness decision and is documented as an assumption. | Set REQ-004 AC-3 and TC-006; a candidate area for future change. |
| DEC-005 | Use **SQLite** for persistence (app and tests). | JSON file persistence. | The exam emphasises *database-backed* tests; SQLite gives real tables, relationships and reproducible seed/reset, and is the recommended option. | Data architecture (D10) uses tables; AT-001..005 run against a reproducible SQLite test DB. |
| DEC-006 | Implement the **RFC tool (CR-01) as its own entity** (`RFC` + `RFCResponse`) at **data + logic + automated-test** level; keep the app UI as the evidence-validation slice. | Build a full RFC screen/flow in the app. | The enunciado requires the app to implement only one small slice; the CR affects requirements/traceability/tests, which are fully covered without a new UI. Modelling RFC as structured, IETF-style documentation (not a chat) satisfies the intent. | Added REQ-011/012, UC-07, US-006, TC-009/010, RFC entities, AT-005; app scope unchanged. |
| DEC-007 | **Reuse the existing role-authorization rule (REQ-008)** so that only the Transition Lead can raise an RFC. | Introduce a separate RFC permission model. | Reusing REQ-008 keeps the security model coherent and avoids duplicate logic; the RFC role check is the same rule as submission. | TC-010 and AT-005 reuse the role logic; REQ-008 updated to cover RFC. |
| DEC-008 | Keep the **academic submission gate** (14 deliverables, dedicated commits, ≥4 days, repo structure) **out of the product Definition of Done**. | Put commit/deliverable rules inside the final-delivery DoD. | The three DoD levels must describe the same object (the AMS product); the process gate is already governed by the commit rules and README. | Final-delivery DoD (D6) rewritten at product level; process tracked separately. |
